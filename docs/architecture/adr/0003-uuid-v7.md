# ADR-0003: UUID v7 默认

## 状态

✅ Accepted (2026-02-01)

## 上下文

AgentID-Chain 每天新增数百万 Agent ID。UUID 选型需考虑：

1. **可排序**：时间范围内 ID 近似单调，便于范围查询
2. **去中心化生成**：客户端可独立生成，无需中心化协调
3. **紧凑性**：尽量小（数据库 / 索引开销）
4. **不可猜测**：避免顺序 ID 被爬取

候选方案：

| 方案 | 长度 | 可排序 | 分布式 | 性能 |
|------|------|--------|--------|------|
| **自增 INT64** | 8B | ✅ | ❌ 需协调 | 优 |
| **UUID v4** | 16B | ❌ | ✅ | 优 |
| **UUID v7** | 16B | ✅（时间序） | ✅ | 优 |
| **ULID** | 16B | ✅ | ✅ | 优 |
| **KSUID** | 20B | ✅ | ✅ | 中 |
| **Snowflake** | 8B | ✅ | ⚠️ 需 worker | 优 |

## 决策

默认采用 **UUID v7**（RFC 9562 草案），保留 v4 作为可选。

理由：
- v7 = 48-bit millisecond timestamp + 74-bit random
- 数据库 B-tree 索引效率接近自增 ID（避免 v4 的随机写入放大）
- 客户端可独立生成，无需 worker / Snowflake-like 协调服务
- 标准 UUID 格式（128-bit），兼容现有工具链

## 后果

### 正面
- ✅ 索引效率高（避免 B-tree 随机写入）
- ✅ 客户端可独立生成（无中心化服务）
- ✅ 时间序自然形成，便于范围查询（`WHERE created_at > X AND uuid > Y`）
- ✅ 不可猜测（74-bit 随机）

### 负面
- ❌ 暴露创建时间（攻击者可推断业务节奏）
- ❌ 需要 Go 1.21+（`uuid.NewV7`）或第三方库

### 中性
- 🔄 v7 不向后兼容 v4（旧系统需迁移）
- 🔄 仍可选用 v4（通过配置）

## 替代方案

| 方案 | 否决理由 |
|------|---------|
| **自增 INT64** | 单点；不可水平扩展；可被爬取 |
| **UUID v4** | 索引效率低（随机写入）；不可排序 |
| **ULID** | 缺乏 UUID 生态兼容性 |
| **Snowflake** | 需 worker；增加运维复杂度 |

## 实现细节

`internal/uuid_generator/`:

```go
// 默认 v7
func New() uuid.UUID {
    return uuid.NewV7()  // Go 1.21+
}

// 可选 v4
func NewV4() uuid.UUID {
    return uuid.New()
}
```

性能基准：205ns/op（见 [uuid-benchmark.md](../../perf/uuid-benchmark.md)）

## 配置

`configs/app.yaml`:
```yaml
uuid:
  version: v7  # v4 | v7
```

## 迁移策略

- v2.0.0 → v2.1.0：默认切换为 v7（不破坏现有 ID）
- 旧 ID 仍为 v4，新 ID 为 v7
- 数据库列类型无需变更（`uuid` 列同时支持）
